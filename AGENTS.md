# AGENTS.md — BrickScan

Conventions and hard-won context for working on this repo. Read this before making changes.

## What this is

BrickScan: SwiftUI iOS app that scans a LEGO set's barcode/box text via the camera (or an
imported photo), looks it up against the Rebrickable catalog, and shows/manages its status in
the user's Rebrickable collection. MVVM + Repository pattern. No third-party dependencies —
Vision, AVFoundation, SwiftUI, SwiftData, URLSession only. Full feature plan: [LEGO_Bricks_App_Plan.md](LEGO_Bricks_App_Plan.md).

## Build & test

The Xcode project is generated by xcodegen from `project.yml` — **never hand-edit
`BrickScan.xcodeproj`** or the generated `BrickScan/Resources/Info.plist`; edit `project.yml`
and regenerate, or your changes get silently overwritten on the next `xcodegen generate`.

```bash
xcodegen generate   # after adding/removing files, or editing project.yml
xcodebuild -project BrickScan.xcodeproj -scheme BrickScan -destination 'generic/platform=iOS Simulator' build
xcrun simctl shutdown all   # run first if a test run fails with "Invalid device state" / Mach errors — simulator flakiness, not a real bug
xcodebuild test -project BrickScan.xcodeproj -scheme BrickScan -destination 'platform=iOS Simulator,name=iPhone 17'
```

Always run both build and test after a change before considering it done. Filter `xcodebuild`
output through `grep -E "error:|Test Suite 'All|** TEST"` — raw output is enormous.

## Architecture

```
BrickScan/
├── App/            BrickScanApp (SwiftData ModelContainer, splash)
├── Core/
│   ├── Network/    NetworkClient, APIError, RebrickableEndpoint (path builders), APIModels
│   ├── Repository/ RebrickableRepository (API calls), LocalRepository (SwiftData cache)
│   ├── Scanner/    BarcodeScanner, OCRScanner (Vision), SetNumberExtractor (regex parsing)
│   └── Storage/    KeychainService, SwiftDataModels (@Model classes)
├── Features/       One folder per screen: Auth (privacy notice/detail only — no login),
│                   Scanner, SetDetail, Settings (API key + account linking), Collection, Splash
└── Tests/BrickScanTests/   MockURLProtocol-based repository tests, SetNumberExtractor tests
```

`RebrickableRepositoryProtocol` is the seam for testing — view models depend on the protocol,
inject `RebrickableRepository()` by default, swap a fake/mock in tests.

## Auth model — read this before touching anything credential-related

There is **no login screen**. The app opens directly into the scanner. Two independent pieces
of Rebrickable auth live in Settings (gear icon):

- **API Key** — static, generated manually on rebrickable.com/profile. Required as
  `Authorization: key {API_KEY}` header on nearly every endpoint. Rebrickable has **no API
  endpoint that derives this from a username/password** — confirmed against the OpenAPI spec.
  Don't try to "simplify" this away again; it's not possible.
- **User token** — obtained via `POST /users/_token/` with username+password, needed in the
  URL path for collection endpoints (`/users/{user_token}/sets/...`). The password is **never
  stored**, discarded immediately after the token is saved. This is done once via the "Lier mon
  compte" section in Settings, not at launch.

If you're asked to add a "login" flow back: don't gate the app behind it. Settings is the right
place for both credentials.

## Rebrickable API — verify shapes before trusting them

`rebrickable.com/api/v3/swagger/?format=openapi` **omits response and form-body schemas** —
it'll show you a path exists but not what it actually accepts/returns. Don't implement against
it blind. Cross-check against the community-maintained, fuller spec:

```bash
git clone --depth 1 https://github.com/rienafairefr/pyrebrickable.git /tmp/pyrebrickable_check
grep -n '"/api/v3/users/{user_token}/sets' /tmp/pyrebrickable_check/rebrickable.json
```

Known gotchas already paid for — don't rediscover them:

- `GET /users/{user_token}/sets/{set_num}/` returns the LegoSet **nested under a `"set"` key**,
  not flat. The spares flag is `include_spares`, not `inc_spares`. (`UserSet` in
  `APIModels.swift` decodes this correctly now — if you touch it, keep the nested shape.)
- `POST /users/{user_token}/sets/` has **no `list_id` parameter** — it auto-picks a list.
  To target a specific list you need `POST /users/{user_token}/setlists/{list_id}/sets/`.
  There is no endpoint to change a set's list directly; "moving" a set means DELETE from the
  old list + POST to the new one. **`RebrickableRepository.addSetToList`/`moveSetToList` still
  use the wrong endpoint as of this writing** — see the open task before relying on them.
- A decode failure surfaces in the UI as `CollectionStatus.unknown("Erreur lors du traitement de
  la réponse")`. If you see that string, suspect a model/JSON-shape mismatch first, not a
  network/auth problem.

## Swift 6 strict concurrency

`SWIFT_STRICT_CONCURRENCY: complete` is on. Patterns already established, keep using them:

- Singletons (`KeychainService.shared`, `NetworkClient.shared`) are `final class X: @unchecked
  Sendable` because their mutable state is private and access is safe by construction (Keychain
  API is thread-safe; NetworkClient holds no mutable state).
- `@Observable` view models that touch UI state are `@MainActor`.
- `@Observable` only tracks **stored** properties. A computed property that reads external state
  (Keychain, disk, etc.) will never trigger a SwiftUI re-render when that state changes — store
  the value and update it explicitly (see `SettingsViewModel.isAccountLinked`, which used to be
  computed and silently never refreshed the UI after unlinking).

## Camera lifecycle

The capture session must stop whenever anything is covering it — sheets, the Photos picker,
Settings. `ScannerView.isMenuOpen` is the single source of truth for this; if you add a new
sheet/picker to `ScannerView`, add its presented-state to `isMenuOpen`, don't add a one-off
`onChange`. `PhotosPicker`'s label-style initializer has no observable presentation state — use
the `.photosPicker(isPresented:selection:matching:)` modifier form instead, as already done.

## Scanning pipeline

`ScannerViewModel.handleFrame` throttles Vision processing to once every 0.8s (`AVCaptureVideoDataOutput`
delivers ~30fps; running barcode+OCR detection on every frame is wasted CPU). A detected
candidate sets `candidateDetected = true` during the 1.5s debounce before the actual API search
— `ScanOverlayView` uses that flag for the pulsing-green/checkmark feedback so the user sees
something happened before the network call starts. Don't remove the throttle or the flag when
touching this file.

## Things deliberately not built (don't re-add without asking)

- No backend caching of collection status — it's always fetched fresh per the original plan
  (avoids showing stale "in collection" state).
- No retry-via-stored-password on a 403 — `RebrickableRepository.reauthenticateAndRefreshToken()`
  always rethrows `.forbidden` by design, since the password is never retained.
